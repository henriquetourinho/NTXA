Segurança Zero Trust:
Protegendo o Organizando.me.

Henrique Tourinho

CTO & Founder @ NXTA

Dados financeiros não são apenas números em um banco de dados. Eles representam o sustento, os sonhos e a estabilidade de uma família. No desenvolvimento do Organizando.me, adotamos uma postura radical: tratar cada dado como se a rede já estivesse comprometida.

O modelo de segurança tradicional (o "castelo com fosso") morreu. Hoje, operamos sob a filosofia de Zero Trust: Nunca confie, sempre verifique.

Criptografia em Repouso (At Rest)

Muitas aplicações criptografam apenas a senha. Isso é insuficiente. Se um atacante obtiver acesso ao banco de dados, ele não deve ser capaz de ler quanto dinheiro você tem ou onde você gastou.

Utilizamos AES-256 para criptografar campos sensíveis (saldos, nomes de transações) antes mesmo de eles tocarem o disco do banco de dados. A chave de descriptografia nunca é armazenada junto com os dados.

Implementação Técnica

Abaixo, um exemplo simplificado da nossa classe de criptografia em Python, utilizando a biblioteca cryptography. Note que a rotação de chaves é mandatória.

security/vault.py 
from cryptography.fernet import Fernet
import os

class FinancialVault:
    def __init__(self):
        # Chave carregada de Variável de Ambiente Segura (Vault)
        self.key = os.getenv('NXTA_MASTER_KEY')
        self.cipher = Fernet(self.key)

    def encrypt_transaction(self, amount: str) -> bytes:
        # Dados são convertidos para bytes antes de cifrar
        return self.cipher.encrypt(amount.encode())

    def decrypt_transaction(self, token: bytes) -> str:
        try:
            return self.cipher.decrypt(token).decode()
        except Exception as e:
            # Log de tentativa de violação de integridade
            SecurityLogger.alert(f"Decryption failed: {e}")
            raise AccessDeniedError()

Autenticação e Sessão

Não usamos JWTs simples. Implementamos Sessões Opacas com rotação automática. Se detectarmos uma mudança anômala de IP ou User-Agent, a sessão é invalidada imediatamente e o usuário deve reautenticar via MFA (Autenticação Multifator).

"Conveniência não pode custar segurança. Preferimos pedir o login novamente a arriscar uma sessão sequestrada."

Auditoria Imutável

Cada ação sensível (transferência, exclusão de conta) gera um log de auditoria assinado criptograficamente. Isso impede que, mesmo um administrador do sistema, altere o histórico de ações para cobrir rastros.

Essa é a engenharia da NXTA: Paranóia construtiva para garantir a paz de espírito dos nossos usuários.
