A Morte do Nginx?
Por que criamos o BrHTTP do zero em Go.

Henrique Tourinho

CTO & Founder @ NXTA

Dependência é dívida técnica. Quando você constrói toda a sua infraestrutura baseada em ferramentas que você não controla, você está alugando o terreno onde constrói seu castelo.

O Nginx é uma peça incrível de engenharia. Ele alimenta a maior parte da web. Mas na NXTA, nosso lema é claro: "A gente não usa. A gente cria."

O Problema do "Canivete Suíço"

Servidores web genéricos como Apache e Nginx são projetados para atender a todos. Eles precisam servir PHP, fazer proxy reverso, balancear carga, servir arquivos estáticos e lidar com milhares de módulos legados.

"Software genérico é otimizado para a média. Nós precisávamos de otimização para a borda."

Precisávamos de algo que integrasse nativamente com nossas soluções de segurança (Criptografia Zero Trust) e que pudesse ser modificado em nível de código para comportar as demandas específicas do DeméterOne (IoT rural) e do BioClock.

A Escolha do Go (Golang)

Rust foi considerado, mas Go venceu pela simplicidade da concorrência. O modelo de Goroutines nos permite lidar com milhares de conexões simultâneas com uma pegada de memória ridícula.

Abaixo, um exemplo simplificado de como o BrHTTP gerencia middlewares de segurança nativa, algo que exigiria configurações complexas de Lua no Nginx:

server/middleware.go 
package main

import (
    "net/http"
    "nxta/security"
)

// SecurityHeader força criptografia proprietária NXTA
func SecurityHeader(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        
        // Rejeita conexões que não são do nosso ecossistema
        if !security.ValidateHandshake(r) {
            http.Error(w, "NXTA: Secure Handshake Required", 403)
            return
        }

        w.Header().Add("X-Server", "BrHTTP/1.0-Alpha")
        w.Header().Add("X-Powered-By", "NXTA Engineering")
        
        next.ServeHTTP(w, r)
    })
}

Performance vs. Controle

Em nossos benchmarks iniciais, o BrHTTP conseguiu servir conteúdo estático apenas 12% mais lento que o Nginx ultra-otimizado. No entanto, em requisições dinâmicas que envolvem nossa lógica de negócio (Agro/Bio), ele foi 40% mais rápido.

Por quê? Porque removemos camadas de abstração. O servidor web é a aplicação. Não há FastCGI, não há overhead de comunicação entre processos.

O Futuro

O BrHTTP não será vendido como um produto isolado, mas é a espinha dorsal que torna o DeméterOne e o Organizando.me tão rápidos e seguros.

Na NXTA, nós descemos até o metal para garantir que o futuro seja construído sobre rocha sólida, não sobre dependências de terceiros.
